## 2024-05-24 - Information Disclosure via Absolute File Paths
**Vulnerability:** Backend API error handlers and data payload extractors were directly embedding full system file paths into error messages (e.g., `SyntaxError` extractions, `Path not found` errors).
**Learning:** Returning full file paths in exceptions or string formats exposes sensitive infrastructure and directory structures to the client, providing attackers with insights into the backend file layout which can be chained for further exploitation.
**Prevention:** Always wrap reflected paths in `os.path.basename()` before formatting them into user-facing output or API responses.
## 2024-05-24 - Zip Slip Vulnerability during Extraction
**Vulnerability:** The zip extraction logic in `app/routers/upload.py` performed path traversal checks using `os.path.commonpath` but did not fully resolve the base path. Attackers could theoretically bypass bounds checking if symlinks were involved or due to improper relative path comparisons when `os.path.realpath` wasn't correctly applied to the entire destination path before checking.
**Learning:** Calling `os.path.realpath` individually on `base_dir` and `safe_filename` then concatenating isn't enough; we must resolve the fully concatenated path. Using `extracted_path.startswith(tmp_dir_real + os.sep)` after calling `os.path.realpath` on the combined string provides a foolproof containment check.
**Prevention:** Always construct the target path first using `os.path.join(base_dir, filename)`, resolve it completely using `os.path.realpath(target_path)`, and then verify that it begins exactly with the resolved `base_dir` string followed by a separator.
## 2024-05-24 - Zip Slip Bypass via Sensitive Hidden Files
**Vulnerability:** The zip extraction logic in `app/routers/upload.py` checked for relative path traversal via `os.path.commonpath` but did not inspect the individual path segments for specifically sensitive files/directories (like `.env`, `.git/`, `.ssh/`). Attackers could exploit this by including a maliciously crafted `.env` or `.git/config` file within an otherwise valid sub-directory of the zip structure, successfully deploying arbitrary configuration.
**Learning:** Checking that a file doesn't escape the temporary directory isn't enough; we must also ensure the contents being extracted do not represent security risks (e.g., hidden config files) when dealing with user-uploaded archives. A blanket ban on all hidden files breaks standard software repositories (which often contain `.gitignore`). We need a specific blocklist.
**Prevention:** Explicitly split paths into segments during zip extraction and compare each segment against a blocklist of sensitive hidden files and directories (like `.env`, `.git`, `.ssh`, `.aws`, `.config`). Reject extraction if any segment matches.
## 2025-04-26 - Insecure File Upload Handling in Multipart
**Vulnerability:** The application properly blocked sensitive hidden files (`.git`, `.env`, etc.) inside zip archive uploads, but failed to block the exact same sensitive names during standard multipart `List[UploadFile]` uploads. `normalize_uploaded_filename` only checked for empty paths and path traversals (`..`), allowing users to upload sensitive files/directories (like `.git/config` or `.env`) into the server's analysis temp directory.
**Learning:** When building defense mechanisms (like sensitive file blocklists) for complex workflows that accept multiple input formats (e.g., zip files AND multipart file uploads), it is critical to apply the boundary checks at a unified, lower-level validation function (`normalize_uploaded_filename`) rather than handling it contextually in only one format's extraction loop.
**Prevention:** Ensure all file paths derived from user input go through a single, comprehensive sanitization utility that applies all business and security rules (traversal checks, absolute path checks, and sensitive name blocklists) before being used in `os.path.join`.

## 2024-05-24 - Rate Limiting on Health & AI Config Endpoints
**Vulnerability:** The `/api/health` and `/api/ai-config` endpoints were missing rate limit protections. Even simple, low-overhead endpoints can be targeted by volumetric DoS attacks to consume thread pool capacity or network bandwidth.
**Learning:** `slowapi` decorators (like `@limiter.limit`) require the decorated FastAPI endpoint function to accept a `request: Request` argument, even if the endpoint logic itself doesn't use the request object, because `slowapi` extracts the client IP and other metadata from it to track rate limits.
**Prevention:** Always verify that every endpoint in a FastAPI application, including simple informational and health checks, is protected by appropriate rate limits and ensures `request: Request` is included in the signature when applying `@limiter.limit`.
## 2025-05-03 - Rate Limit Bypass via IP Spoofing behind Reverse Proxies
**Vulnerability:** Fast API applications using `slowapi` rate limiting behind reverse proxies (e.g., Fly.io) track rate limits based on the IP address. Without proper middleware to parse `X-Forwarded-For` headers correctly while verifying trusted proxies, attackers can spoof their IP address to bypass rate limits entirely or exhaust limits for legitimate users.
**Learning:** `ProxyHeadersMiddleware` must be explicitly added to properly rewrite the client IP address from proxy headers. Due to FastAPI's LIFO middleware execution, `ProxyHeadersMiddleware` must be added *after* `SlowAPIMiddleware` so it executes *before* the rate limiter evaluates the client's IP. Additionally, it must be constrained to a trusted proxy list (e.g., via env vars) instead of globally trusting `*`.
**Prevention:** Use `uvicorn.middleware.proxy_headers.ProxyHeadersMiddleware` with a strictly controlled `trusted_hosts` configuration. Add it to the FastAPI application specifically after `SlowAPIMiddleware` (which translates to running before it in the stack) to ensure IPs are rewritten before rate limits are checked.
## 2025-05-06 - Insecure CORS Configuration Wildcard Bypass
**Vulnerability:** The CORS configuration in `app/__init__.py` was directly parsing the `VIBEGRAPH_CORS_ORIGINS` environment variable using `.split(",")` without stripping whitespace or validating the origins. This allowed an attacker to bypass the "allow_credentials" logic by providing a space-padded wildcard (e.g., ` * `), which bypasses the strict `"*"` check but is often treated as a valid wildcard by browsers, allowing cross-origin credentialed requests.
**Learning:** Raw string splitting of comma-separated environment variables often leaves leading/trailing whitespaces. When building security controls (like CORS configurations), an exact string match (like `allow_credentials = "*" not in origins_list`) can be bypassed if the inputs are not properly normalized and validated against an allowed pattern.
**Prevention:** Always sanitize CORS origins by stripping whitespace and filtering out empty strings. Validate that each origin is either exactly `*` or starts with a valid scheme like `http://` or `https://`. Reject and log any invalid origins.

## 2025-05-11 - Zip Extraction Path Traversal and Sensitive File Bypass
**Vulnerability:** The ZIP extraction logic in `app/routers/upload.py` implemented custom, inline path traversal and sensitive file checks that differed from the application's standard file upload validation. This could potentially allow bypasses if edge cases were missed. Additionally, the standard normalizer (`normalize_uploaded_filename`) can be misleading if its return value is ignored.
**Learning:** Security validation logic must be unified across all upload paths. Implementing separate inline checks for ZIP members creates an inconsistent security boundary. Furthermore, when using a normalizer, it is critical to use its *return value* as the finalized path. Calling a normalizer but continuing to use the raw unsanitized input path completely bypasses the intended security sanitizations.
**Prevention:** Unify validation logic by always calling the core `normalize_uploaded_filename()` utility on extracted ZIP members, and explicitly using its return value (`safe_filename = normalize_uploaded_filename(...)`) for all subsequent path operations to enforce a single, robust security boundary.

## 2025-05-24 - Information Disclosure via Arbitrary File Read in cwd
**Vulnerability:** The `is_safe_path` function in `app/utils/security.py` intentionally allowed read access to any non-hidden file located in the server's current working directory (`cwd`). In a hosted or public-facing environment, this constitutes an Information Disclosure vulnerability because attackers could use code analysis endpoints (`/api/snippet`, `/api/explain`, `/api/learning-path`) to read the server's own source code, dependencies, and configuration files.
**Learning:** Security validation functions designed for file access must strictly enforce boundaries to only temporary or explicitly designated user-uploaded directories. Falling back to the application's root execution directory (`cwd`) bypasses isolation, exposing the application's intellectual property and environmental context.
**Prevention:** Strictly enforce that `is_safe_path` only allows paths that reside within the designated temporary upload directory boundaries (e.g., matching `UPLOAD_PREFIX` or `vibegraph_test_` within the system's temp directory) and completely reject attempts to resolve paths to the executing `cwd`.
## 2024-05-13 - Prevent Zip Symlink Path Traversal
**Vulnerability:** Zip extraction code (`zip_ref.open(member)`) failed to check if a member was a symbolic link. An attacker could craft a ZIP archive containing a symlink pointing to an arbitrary location (e.g., `/etc/passwd`), followed by a file targeting that symlink, resulting in arbitrary file overwrite (Zip Slip / Path Traversal) via symlinks.
**Learning:** Python's `zipfile.ZipInfo` object does not have a built-in `is_symlink()` method. To check if a member is a symlink, you must use `stat.S_ISLNK(member.external_attr >> 16)`. Temporary scratch files created during testing or investigation must be cleaned up before committing to avoid polluting the codebase.
**Prevention:** Check for symlinks during ZIP extraction using `stat.S_ISLNK(member.external_attr >> 16)` and abort the extraction/raise an error if detected.
