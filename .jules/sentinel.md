## 2025-05-10 - Unifying Path Bounds Checking
**Vulnerability:** When building defense mechanisms (like sensitive file blocklists) for complex workflows that accept multiple input formats (e.g., zip files AND multipart file uploads), it is critical to apply the boundary checks at a unified, lower-level validation function (`normalize_uploaded_filename`) rather than handling it contextually in only one format's extraction loop.
**Learning:** Security validations using a whitelist must be strictly enforced. If a configuration parameter (like a strict domain list) indicates a strict security posture is desired, explicitly allowing a permissive fallback (like a wildcard) creates a critical bypass.
**Prevention:** Ensure all file paths derived from user input go through a single, comprehensive sanitization utility that applies all business and security rules (traversal checks, absolute path checks, and sensitive name blocklists) before being used in `os.path.join`.

## 2024-05-24 - Rate Limiting on Health & AI Config Endpoints
**Vulnerability:** The `/api/health` and `/api/ai-config` endpoints were missing rate limit protections. Even simple, low-overhead endpoints can be targeted by volumetric DoS attacks to consume thread pool capacity or network bandwidth.
**Learning:** `slowapi` decorators (like `@limiter.limit`) require the decorated FastAPI endpoint function to accept a `request: Request` argument, even if the endpoint logic itself doesn't use the request object, because `slowapi` extracts the client IP and other metadata from it to track rate limits.
**Prevention:** Always verify that every endpoint in a FastAPI application, including simple informational and health checks, is protected by appropriate rate limits and ensures `request: Request` is included in the signature when applying `@limiter.limit`.

## 2025-05-03 - Rate Limit Bypass via IP Spoofing behind Reverse Proxies
**Vulnerability:** Fast API applications using `slowapi` rate limiting behind reverse proxies (e.g., Fly.io) track rate limits based on the IP address. Without proper middleware to parse `X-Forwarded-For` headers correctly while verifying trusted proxies, attackers can spoof their IP address to bypass rate limits entirely or exhaust limits for legitimate users.
**Learning:** `ProxyHeadersMiddleware` must be explicitly added to properly rewrite the client IP address from proxy headers. Due to FastAPI's LIFO middleware execution, `ProxyHeadersMiddleware` must be added *after* `SlowAPIMiddleware` so it executes *before* the rate limiter evaluates the client's IP. Additionally, it must be constrained to a trusted proxy list (e.g., via env vars) instead of globally trusting `*`.
**Prevention:** Use `uvicorn.middleware.proxy_headers.ProxyHeadersMiddleware` with a strictly controlled `trusted_hosts` configuration. Add it to the FastAPI application specifically after `SlowAPIMiddleware` (which translates to running before it in the stack) to ensure IPs are rewritten before rate limits are checked.

## 2025-05-06 - Insecure CORS Configuration Wildcard Bypass
**Vulnerability:** The CORS configuration in `app/__init__.py` was directly parsing the `VIBEGRAPH_CORS_ORIGINS` environment variable using `.split(",")` without stripping whitespace or validating the origins. This allowed an attacker to bypass the "allow_credentials" logic by providing a space-padded wildcard (e.g., ` * `), which bypasses the strict `"*"` check but is often treated as a valid wildcard by browsers, allowing cross-origin credentialed requests.
**Learning:** Raw string splitting of comma-separated environment variables often leaves leading/trailing whitespaces. When building security controls (like CORS configurations), an exact string match (like `allow_credentials = "*" not in origins_list`) can be bypassed if the inputs are not properly normalized and validated against an allowed pattern.
**Prevention:** Always sanitize CORS origins by stripping whitespace and filtering out empty strings. Validate that each origin is either exactly `*` or starts with a valid scheme like `http://` or `https://`. Reject and log any invalid origins.

## 2025-05-11 - Zip Extraction Path Traversal and Sensitive File Bypass
**Vulnerability:** The ZIP extraction logic in `app/routers/upload.py` implemented custom, inline path traversal and sensitive file checks that differed from the application's standard file upload validation. This could potentially allow bypasses if edge cases were missed. Additionally, the standard normalizer (`normalize_uploaded_filename`) can be misleading if its return value is ignored.
**Learning:** Security validation logic must be unified across all upload paths. Implementing separate inline checks for ZIP members creates an inconsistent security boundary. Furthermore, when using a normalizer, it is critical to use its *return value* as the finalized path. Calling a normalizer but continuing to use the raw unsanitized input path completely bypasses the intended security sanitizations.
**Prevention:** Unify validation logic by always calling the core `normalize_uploaded_filename()` utility on extracted ZIP members, and explicitly using its return value (`safe_filename = normalize_uploaded_filename(...)`) for all subsequent path operations to enforce a single, robust security boundary.

## 2025-05-24 - Missing Input Sanitization in LLM Prompts
**Vulnerability:** The `ChatRequest` Pydantic model lacked input validation for the `node_id` and `file_path` fields, potentially allowing prompt injection attacks when these values are passed to the backend LLM. Also, redefining identical `@field_validator` classmethod names across multiple models inside the same file (e.g., `sanitize_identifiers`) causes F811 `Redefinition of unused name` linting errors in Ruff.
**Learning:** Shared validator functions across Pydantic models in the same file must have unique names to avoid linting errors. Additionally, `check_fields=False` is critical when a generic validator is shared via classmethods or mixins across models where some fields may not exist.
**Prevention:** Apply input sanitization to all parameters passed into LLM prompts. Ensure validator methods have uniquely assigned names per model within the same file and use `check_fields=False` when appropriate.

## 2025-05-24 - Missing Input Sanitization Truncation
**Vulnerability:** Identifier fields passed to `sanitize_llm_input` in `app/models.py` did not explicitly enable truncation before applying a regex filter. Because Pydantic checks `max_length` constraints after `mode="before"` field validators, an attacker could supply an extremely long string (e.g. 100MB) that would be processed by the regex, leading to CPU exhaustion / ReDoS.
**Learning:** Pydantic validators with `mode="before"` are executed before any built-in `max_length` limits. Security sanitization functions that employ regex MUST enforce their own length bounds before evaluating the regex to prevent Denial of Service attacks.
**Prevention:** Always pass `truncate=True` and a reasonable `max_length` to `sanitize_llm_input` when using it as a `mode="before"` validator on user-provided identifiers, so that bounded inputs are processed by the regex and subsequently cleanly rejected by Pydantic if they were maliciously large. Do not apply this redundantly to fields that already manually validate their length with `raise ValueError` before the regex.

## 2025-06-03 - Asymmetric DoS via Massive Multipart Form Uploads
**Vulnerability:** The `upload_project` endpoint in `app/routers/upload.py` strictly limited the number of files inside a `.zip` archive via `MAX_ZIP_FILES = 10000`, but failed to enforce a corresponding parity limit on raw `List[UploadFile]` multipart uploads. An attacker could bypass the zip protections and exhaust server memory, inodes, or processing power by submitting a massive multipart array containing tens of thousands of tiny files.
**Learning:** Security validations on multi-format upload endpoints (like `.zip` and multipart forms) must enforce identical threshold constraints. A limit protecting one ingestion path does not protect the other.
**Prevention:** Enforce an explicit application-layer `MAX_UPLOAD_FILES` bound on the length of `List[UploadFile]` array prior to processing, maintaining parity with `.zip` extraction limits. Note: while framework layers (like Starlette's `max_files` default of 1000) may offer baseline protection, relying exclusively on undocumented or easily altered framework defaults instead of explicit business logic is fragile.

## 2025-06-05 - Pydantic Validator Redefinition Masking
**Vulnerability:** Multiple Pydantic models in `app/models.py` repeatedly reused the same `@field_validator` classmethod names (e.g., `normalize_model`, `validate_node_context_lists`). This identical naming triggers Pydantic `UserWarning`s ("overrides an existing Pydantic @field_validator decorator") and Ruff `F811 Redefinition of unused name` errors. In large codebases, these silenced or ignored linting errors can silently mask missing or incorrectly overridden security validation checks, allowing input validation bypasses.
**Learning:** Each `@field_validator` assigned to different fields or models in the same file must have a unique method name. Pydantic maps validators heavily based on the function name itself. Reusing names overrides the global namespace and suppresses critical strictness regarding what fields are actually being validated.
**Prevention:** Always assign unique names to `@field_validator` methods across all Pydantic models within the same module (e.g., `normalize_model_chat`, `normalize_model_learning`). This ensures every validator functions strictly for the fields and models it was intended for without bleeding over or silencing tooling alerts.
