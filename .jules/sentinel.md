## 2024-05-24 - Information Disclosure via Absolute File Paths
**Vulnerability:** Backend API error handlers and data payload extractors were directly embedding full system file paths into error messages (e.g., `SyntaxError` extractions, `Path not found` errors).
**Learning:** Returning full file paths in exceptions or string formats exposes sensitive infrastructure and directory structures to the client, providing attackers with insights into the backend file layout which can be chained for further exploitation.
**Prevention:** Always wrap reflected paths in `os.path.basename()` before formatting them into user-facing output or API responses.
## 2024-05-24 - Zip Slip Vulnerability during Extraction
**Vulnerability:** The zip extraction logic in `app/routers/upload.py` performed path traversal checks using `os.path.commonpath` but did not fully resolve the base path. Attackers could theoretically bypass bounds checking if symlinks were involved or due to improper relative path comparisons when `os.path.realpath` wasn't correctly applied to the entire destination path before checking.
**Learning:** Calling `os.path.realpath` individually on `base_dir` and `safe_filename` then concatenating isn't enough; we must resolve the fully concatenated path. Using `extracted_path.startswith(tmp_dir_real + os.sep)` after calling `os.path.realpath` on the combined string provides a foolproof containment check.
**Prevention:** Always construct the target path first using `os.path.join(base_dir, filename)`, resolve it completely using `os.path.realpath(target_path)`, and then verify that it begins exactly with the resolved `base_dir` string followed by a separator.
## 2024-05-24 - Zip Storage Exhaustion DoS vulnerability via Local Variables
**Vulnerability:** In `app/routers/upload.py`, a `total_size` accumulator to protect against unbounded storage exhaustion/DoS attacks during zip extraction was reset to `0` at the start of each individual file extraction or iteration. When multiple zip files were uploaded in a single request, the size limits would effectively be per-file rather than global across the entire upload payload, potentially allowing DoS via massive multi-file uploads bypassing the `MAX_UNCOMPRESSED_SIZE` limits. Also, the same `total_size` variable was used for header size checks and chunk streaming extraction which double-counted the byte length.
**Learning:** Initializing accumulators inside loop blocks (such as `for file in files:`) negates global limit enforcement across a request payload. Additionally, using a single accumulator for multiple distinct checks (e.g. zip headers and streaming extraction chunks) leads to inaccurate accumulation that either over/under counts the limits.
**Prevention:** Global request limits for resource-intensive operations (such as multi-file and uncompressed zip operations) must initialize accumulator variables at the start of the request handling block, outside of any iterative loops. Separate accumulators should be used for different distinct counting operations like extraction streaming and zip headers.
